类加载机制

0. 探讨背景

在使用jstorm开发的过程中，遇到了jar包冲突的问题，就是当把打包好的jar包执行之后，集群会报错，而在本地跑时确是正常的，这时候面临两个解决方案。

  1. 配置classloader参数。topology.enable.classloader: false, 默认关闭classloader，如果应用的jar与JStorm的依赖的jar发生冲突，比如应用使用thrift9，但jstorm使用thrift7时，就需要打开classloader。建议在集群级别上默认关闭，在具体需要隔离的topology上打开这个选项。
  2. 用maven解决冲突。解决办法是讲版本冲突的jar包抠掉，调试直到成功为止。使用exclude
    项目使用了第二种方式，解决的办法之后会文章记录，本文是处于对第一种解决办法的好奇而开始研究的。

1. java类加载基本原理

系统提供的类加载器

Java 中的类加载器大致可以分成两类，一类是系统提供的，另外一类则是由 Java 应用开发人员编写的。系统提供的类加载器主要有下面三个：

  • 引导类加载器（bootstrap class loader）：它用来加载 Java 的核心库，是用原生代码来实现的，并不继承自 java.lang.ClassLoader。加载rt.jar和jre/lib 下面一下重要的类

  • 扩展类加载器（extensions class loader）：它用来加载 Java 的扩展库。Java 虚拟机的实现会提供一个扩展库目录。该类加载器在此目录里面查找并加载 Java 类。默认加载目录 jre/lib/ext 下面的jar包

  • 系统类加载器（system class loader）：它根据 Java 应用的类路径（CLASSPATH）来加载 Java 类。一般来说，Java 应用的类都是由它来完成加载的。可以通过 ClassLoader.getSystemClassLoader()来获取它。
